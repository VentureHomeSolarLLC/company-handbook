SUBJECT:
Solving the Shared Credentials Problem (ConEd, PSEG, & Sunnova)

1. The Shared Credentials Problem

In the Agora, multiple different Engrams will need to access the same external systems.

  • Interconnection Agent: Needs ConEd login to submit applications.
  • PTO Agent: Needs ConEd login to download approval letters.
  • Performance Agent: Needs ConEd login to pull meter data.

If we hardcode username: "ops@venturehome.com", password: "Solar2026!" into every single SKILL.md file, we create a security nightmare. If the password changes, we have to update 50 files. If an agent is compromised, the plain-text password is exposed.

2. The Solution: The Keychain Architecture

We solve this by separating Reference Data (URLs, Login Page Locations) from Secrets (Usernames, Passwords).

Component A: The Public Concept (The Map)

The Concept File (`engrams/interconnection/concepts/utilities/coned.md`) stores the "Public" information about how to access the system.

# Con Edison (ConEd) - **Login URL:** https://www.coned.com/login - **Credential Key:** `CONED_OPS_USER` - **Diagram Requirement:** 3-Line

Component B: The Secure Keychain (The Keys)

The actual credentials are stored in the Agent's Environment Variables (`openclaw.json` or `.env`). This is the secure "Keychain" that the agent carries.

"env": { "CONED_OPS_USER": "ops@venturehome.com", "CONED_OPS_PASS": "Solar2026!", "PSEG_OPS_USER": "interconnect@venturehome.com" }

3. How the Agent Uses the Keys

When an agent executes a Skill, it references the credential by its Key, never by its value.

# SKILL: Submit Application 1. **Load Concept:** Read `concepts/utilities/coned.md`. -> Agent learns the URL is `coned.com` and the Key is `CONED_OPS_USER`. 2. **Retrieve Secret:** The Agent queries its own environment for the value of `CONED_OPS_USER`. -> System returns: "ops@venturehome.com" (The Agent holds this in temporary memory only). 3. **Execute Login:** The Agent uses the `browser` tool to inject the credentials. -> `browser(action="type", selector="#username", text=process.env.CONED_OPS_USER)`

4. Why This is Secure & Scalable

Feature Benefit
Centralized Updates If we change the ConEd password, we update it in one place (the central config). All 50 agents instantly use the new password without code changes.
No Hardcoded Secrets Our GitHub repositories (Engrams) contain zero passwords. We can safely share Engrams with contractors or new hires without leaking root access.
Role-Based Access We can give different agents different keys. The "Sales Agent" might have the SALES_CRM_USER key, while the "Ops Agent" has the ADMIN_CRM_USER key. The Skill code remains the same; only the keys change.

5. Summary

To build a secure Agora:

  1. Engrams tell the agent where to log in.
  2. Skills tell the agent how to log in.
  3. Environment Variables provide the keys to log in.
Security Rule #1

NEVER commit a password to a Markdown file, JSON file, or Script inside an Engram. Always use an Environment Variable reference (e.g., $CONED_PASSWORD).
